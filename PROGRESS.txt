Phase 4 — Dedicated Server & Live Multiplayer
  Authoritative server (clients send inputs; server runs all physics)
  TCP game server on Raspberry Pi 4
  Live 1v1 real-time matches
  Take a Turn (TAT) async matches
  Version handshake (client/server must match exactly)
  Reconnect window (3-minute grace, both casual and ranked)
  Opponent quit notification
  OTA auto-update on launch and before multiplayer
  IRC channel dashboard at /ircdash/ (live chat, ZPG/DiceRPG leaderboards)
  IRC channels: #lobby, #general, #dicerpg, #zpg (renamed from #General-Chat → #general)
  Parity system — three enforcement layers:
    1. Compile-time: _gamestate_parity_checklist forces field classification
    2. Architecture: aim_angle flows through server_tick; Up/Down never stripped;
       new cursor-phase weapons are automatic in all 5 paths
    3. Runtime: assert_all_paths_in_sync in tests/parity.rs covers all 5 paths

KEY ARCHITECTURE NOTES
-----------------------
  simulate_with_muzzle(game, input, muzzle, aim_angle) — single sim function
    called by all 5 paths. aim_angle: Option<f32> lets server pass received angle
    directly; Up/Down buttons are never stripped and flow to all weapons.
  StateMsg round-trip — live client rebuilds state from server's StateMsg each tick.
    Never runs simulate. Enforced by compile-time checklists in net_sync.rs.
  emit_fx / emit_sound — all particle FX and sounds go through these channels
    to auto-replicate to live clients. Never call fx functions directly from sim.
  assert_all_paths_in_sync — parity test helper in tests/parity.rs. Call for
    every new gameplay feature test.
